Airline Ticketing
Cloud Migration

Microsoft Azure Python Bash Oracle SQL Jenkins Kubernetes OpenShift Splunk Grafana MinIO Podman Control-M
Context

Progressive migration of a multi-airline ticketing platform from on-premise to Microsoft Azure — requiring strong observability, and test reliability.

What I did
01 · SRE / Incidents
Production L2/L3 response
Resolved recurring OOM caused by Kafka lag via memory limit tuning. Built a controlled skip mechanism for partition-blocking orders. Deep-traced a Splunk missing-index failure causing endless EDA retries, cross-reference issues between a European airline and a Gulf carrier airline systems, and incorrect HTTP paths on deployed microservices.
02 · CI/CD
Pipeline reliability fixes
Fixed pod reuse memory leaks by enforcing fresh agent creation per build phase. Added per-package Helm retry logic against transient network failures. Fixed PR build-status logic to always consider last result. Automated PR job creation before scan to eliminate false-success merges.
03 · Automation
CI/CD promotion & Teams alerts
Designed a single-PR workflow — developer submits one PR for the first environment only. Pipeline auto-promotes across environments, creates PRs per phase, and posts real-time build status (creation, success, failure, merge) into the exact Teams thread via Power Automate.
04 · Governance
Audit log drift detection
Detected manual env variable, feature toggle, and version changes on customer-facing OpenShift systems. 05 · Migration
Kafka & MongoDB migrations

European airline MongoDB: resolved 100% CPU saturation by reducing simultaneous collection injections and adding time delays between document batches.

Identified replica count desync with Custom Resource, duplicate Kafka Topic Operator object generation, and config file bloat impacting readability. Mission 03

Vodafone TV STB &
VSB — CI/CD Infrastructure

Ansible Apache HTTP Server Bash Buildroot Debian Docker Jenkins GitLab CI Grafana Prometheus Nexus Python
Context

CI/CD infrastructure for Set-Top Box (STB) and Video Service Box (VSB) embedded software at Vodafone. A 12-developer team running Scrum sprints built firmware for physical TV boxes — each compilation taking 5–6 hours. The environment had no standardisation, no observability, and builds were routinely killed by network connectivity failures mid-way through with no resume capability.

Challenges
  • Kerberos authentication to GitLab expired mid-compilation — network failures killed 5–6 hour builds with no resume capability, wasting full days of compute
  • Binary security signing and Nexus artifact upload were manual, slow, error-prone final release steps
  • No environment standardisation — Jenkins, Nexus, and STBTester ran differently on each of 5 servers
  • No observability across 16 CI pipelines and 5 servers — no way to detect failures, performance degradation, or infrastructure drift
What I did
01 · Reliability
Kerberos network auto-recovery
Deployed a Keytab script + Kerberos config via Ansible across the Jenkins cluster. The script runs periodically, checks GitLab connectivity, and automatically renews Kerberos authentication if it fails. Builds resume from where they stopped with a 3-retry mechanism for external dependencies — eliminating all wasted 5–6 hour compilation losses.
02 · CI/CD pipeline
End-to-end build & test pipeline
PR created → GitLab CI check → Jenkins job triggered after merge → 5–6 hr Buildroot compilation → binary upload to Nexus → install on test STB/VSB hardware → NR tests executed → HTML test report auto-generated and delivered by email to all stakeholders.
03 · Automation
Binary signing & release automation
Bash scripts automating binary security signing and Nexus artifact upload — previously a manual multi-step release process. Reduced total release execution time by 92%.
04 · Infrastructure
Containerised dev infrastructure
Containerised STBTester, Jenkins, and Nexus using Docker; deployed and configured across all servers via Ansible. Eliminated environment divergence and allowed infrastructure to be reproduced or rebuilt from code.
05 · Observability
Grafana/Prometheus dashboards
Built Grafana/Prometheus dashboards monitoring 16 CI pipelines and 5 servers in real time. Python script auto-generates structured HTML compilation reports delivered by email — giving the 12-dev team instant visibility into build status, failure patterns, and infrastructure health.
